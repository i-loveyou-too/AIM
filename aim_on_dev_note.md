# Aim ON 개발 노트

## 0. 문서 개요

## 개발 대원칙

### 1. 잘 돌아가는 기능은 함부로 건드리지 않기

* 현재 정상 동작하는 기능은 이유 없이 수정하지 않기
* 수정이 필요할 때는 정확한 목적이 있을 때만 건드리기
* 단순히 "더 예쁠 것 같아서", "왠지 이게 더 좋아 보여서" 수준이면 보류하기

### 2. 수정은 최소 범위로 하기

* 한 번에 여러 군데를 동시에 바꾸지 않기
* 필요한 파일과 필요한 부분만 최소 수정하기
* 큰 리팩토링보다 작은 수정으로 해결 가능한지 먼저 보기

### 3. 한 번에 하나씩 고치기

* 버그 수정, UI 수정, 구조 변경을 한 번에 몰아서 하지 않기
* 한 작업 단위마다 결과 확인 후 다음 단계로 넘어가기
* 무엇 때문에 깨졌는지 추적 가능하게 작업하기

### 4. 바꾸기 전에 현재 상태를 기록하기

* 수정 전 어떤 화면/기능이 어떻게 동작했는지 간단히 적기
* 큰 수정 전에는 파일명, 핵심 변경 포인트, 기대 결과를 기록하기
* 가능하면 수정 전 화면 캡처나 메모 남기기

### 5. 모든 중요한 변경은 개발 노트에 기록하기

* 무엇을 바꿨는지
* 왜 바꿨는지
* 어떤 파일을 건드렸는지
* 결과가 어땠는지
* 다음에 이어서 무엇을 해야 하는지

### 6. 추측으로 수정하지 않기

* 원인도 모른 채 이것저것 다 건드리지 않기
* 문제 증상, 원인 추정, 확인 결과를 나눠서 생각하기
* 확실하지 않으면 먼저 구조를 읽고 이해한 뒤 수정하기

### 7. UI보다 동작 안정성을 우선하기

* 예쁜 것보다 안 깨지는 것이 먼저
* MVP 단계에서는 완벽한 미감보다 사용 흐름이 정상 동작하는 것이 우선
* 화면 개선은 기능이 안정된 뒤 진행하기

### 8. 구조를 먼저 보고 코드를 쓰기

* 새 기능을 넣기 전에 기존 구조와 충돌하는지 확인하기
* 비슷한 코드가 이미 있으면 재사용 가능한지 먼저 보기
* 중복 코드를 급하게 늘리지 않기

### 9. 임시 코드와 확정 코드를 구분하기

* 임시 처리한 부분은 반드시 표시하기
* mock data, 임시 버튼, 임시 문구는 나중에 정리할 수 있게 따로 적기
* 지금 되는 것과 나중에 고도화할 것을 분리해서 관리하기

### 10. 에러가 나면 되돌릴 수 있게 작업하기

* 큰 수정 전에는 복구 가능한 상태를 남기기
* 어디까지가 마지막 정상 상태였는지 기억하기
* 한 번에 너무 많이 바꿔서 되돌리기 어렵게 만들지 않기

### 11. 화면별 책임을 섞지 않기

* 로그인, 학생관리, 과제관리, 학습계획의 역할을 섞지 않기
* 한 페이지는 한 핵심 목적에 집중하게 만들기
* 페이지가 복잡해지면 기능을 분리하는 방향 먼저 검토하기

### 12. 지금 필요한 것만 만들기

* 나중에 필요할 것까지 한 번에 다 만들려 하지 않기
* 현재 MVP 목표에 직접 필요한 기능부터 만들기
* 미래 확장은 고려하되, 현재 범위를 넘는 구현은 미루기

### 13. 문서와 실제 구현을 최대한 맞추기

* 문서에 적힌 구조와 실제 코드가 너무 다르지 않게 유지하기
* 방향이 바뀌면 문서를 같이 업데이트하기
* 개발 노트는 형식용이 아니라 실제 작업 기준 문서로 쓰기

### 14. 사람이 읽기 쉬운 코드를 우선하기

* 나중에 다시 봤을 때 바로 이해되는 코드로 작성하기
* 변수명, 파일명, 템플릿명은 직관적으로 짓기
* 너무 똑똑한 코드보다 명확한 코드가 우선
* 복잡한 코드 블록에는 역할이 드러나는 짧은 주석이나 각주를 남기기

### 15. 문제를 해결했으면 재발 방지 메모 남기기

* 같은 문제를 또 겪지 않게 원인과 해결 포인트를 짧게라도 남기기
* 특히 경로, 한글 깨짐, 템플릿 상속, static 연결 문제는 반복 기록하기

### 16. 파일은 기능 기준으로 쪼개기

* 파일 분리는 "페이지 기준 + 역할 기준"으로 하기
* 하나의 파일이 여러 페이지 책임까지 떠안지 않게 하기
* 한 파일이 너무 길어지거나, 여러 종류의 일을 같이 하면 분리 대상이라고 보기

### 17. 파일 분리의 목적은 예쁨이 아니라 유지보수

* 쪼개는 이유는 나중에 찾기 쉽고, 수정 범위를 줄이고, 오류 전파를 막기 위해서임
* 처음부터 과하게 잘게 쪼개지는 않되, 한 파일이 비대해지면 바로 나누기

### 18. 공통과 개별을 분리하기

* 공통 레이아웃, 공통 스타일, 공통 스크립트는 따로 두기
* 페이지 전용 코드는 해당 페이지 파일로 분리하기
* 한 페이지에서만 쓰는 코드는 공통 파일에 넣지 않기

## 7. 기술 스택

### 현재 선택

* Frontend: Next.js + TypeScript + Tailwind CSS + shadcn/ui
* Backend: Django + Django REST Framework

### 선택 이유

* 프론트엔드와 백엔드 책임을 깔끔하게 분리할 수 있음
* UI는 빠르게 만들면서도 일관된 디자인 시스템을 유지하기 좋음
* Django는 운영형 데이터와 관리자 기능에 강함
* DRF를 붙이면 나중에 학생용 서비스나 홍보 사이트 확장도 수월함

### 나중에 검토 가능

* 인증 방식 구체화
* API 버전 관리
* 상태 관리 방식 정리
* 학생용 페이지 및 앱화 확장

### 메모

* 초기에는 mock data 중심
* template 분리 철저히
* base layout 재사용
* view / service / mock_data를 역할별로 분리
* 페이지가 늘어나도 같은 기준으로 확장 가능하게 유지

### 파일 분리 기준

#### 1. 페이지가 다르면 파일도 분리

* 로그인, 학생관리, 과제관리, 학습계획은 각각 별도 템플릿과 별도 스타일을 기본으로 함
* 서로 다른 화면 목적을 하나의 큰 파일에 우겨넣지 않기

#### 2. 공통 요소는 component로 분리

* 카드, 뱃지, 테이블 일부, 헤더, 사이드바처럼 여러 곳에서 반복되는 것은 component로 분리
* 단 한 페이지에서만 쓰는 구조면 굳이 공통 component로 빼지 않기

#### 3. CSS는 공통 / 레이아웃 / 페이지 전용으로 분리

* base.css: 폰트, 색상 변수, 리셋, 기본 버튼 등
* layout.css: 사이드바, 헤더, 공통 레이아웃
* components.css: 카드, 뱃지, 테이블, 공통 UI 조각
* pages/*.css: 각 페이지에서만 쓰는 스타일

#### 4. JS는 페이지별로 분리

* 전체 공통 동작만 main.js에 두기
* 검색, 필터, 토글처럼 특정 페이지에서만 쓰는 동작은 해당 페이지 js에 넣기
* 모든 스크립트를 main.js 하나에 몰지 않기

#### 5. View도 기능별로 분리

* student 관련 view는 student_views.py
* assignment 관련 view는 assignment_views.py
* dashboard 관련 view는 dashboard_views.py
* 기능이 다른데 같은 파일에 쌓이지 않게 유지

#### 6. 데이터 준비 로직도 분리

* mock data나 가공 로직은 view 안에 길게 쓰지 않기
* 서비스 파일이나 mock_data 파일로 분리해서 view는 최대한 짧게 유지

#### 7. 한 파일이 너무 길어지면 분리 신호로 보기

* HTML/CSS/JS/Python 어느 파일이든 길어지고 역할이 두 개 이상 섞이면 나누기
* 특히 페이지 템플릿이 너무 길어지면 component 추출 먼저 검토

#### 8. 찾기 쉬운 이름으로 저장

* students.html, student_detail.html 같이 역할이 바로 보이게 이름 짓기
* style1.css, final.js 같은 이름 금지

#### 9. 아직 한 번만 쓰는 코드라도 곧 반복될 것 같으면 미리 분리 가능

* 단, 미래를 과하게 상상해서 처음부터 너무 쪼개지는 않기
* 반복 가능성이 명확할 때만 선제 분리

#### 10. 분리 기준의 핵심

* 어디를 수정해야 하는지 바로 찾을 수 있어야 함
* 한 기능 수정이 다른 화면에 피해를 덜 줘야 함
* Claude나 다른 도구에게 특정 파일만 건드리라고 지시하기 쉬워야 함
