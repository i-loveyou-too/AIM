# DB 데이터 요구사항 정리

## 1. 문서 목적

이 문서는 Aim ON 프로젝트의 DB 테이블을 실제로 설계하기 전에, 선생님 1명과 학생 1명 기준으로 어떤 데이터를 저장해야 하는지 정리하기 위한 요구사항 문서다.

주요 목적은 다음과 같다.

- `users`, `teacher_profiles`, `academies`, `academy_members`, `students`, `enrollments` 등 핵심 테이블 설계 전에 저장 대상 데이터를 합의한다.
- 화면 기획이 아니라 데이터 저장 단위 기준으로 필수 정보와 선택 정보를 구분한다.
- MVP에서 지금 바로 넣을 컬럼과, 운영 후 확장해도 되는 컬럼을 분리한다.
- 개인정보와 민감정보를 구분하여 최소 수집 원칙을 지킨다.

이 문서는 MVP 기준 문서이며, 이후 실제 ERD 작성과 migration 설계의 기준 문서로 사용한다.

## 2. 공통 원칙

- 필수 / 선택 / 추후 확장으로 구분한다.
- 개인정보는 서비스 운영에 꼭 필요한 최소 항목만 수집한다.
- 로그인 정보와 프로필 정보는 분리한다.
  - 로그인 정보: 인증, 계정 상태, 마지막 로그인
  - 프로필 정보: 표시명, 담당 과목, 소개, 소속 정보
- 학원 소속과 권한은 별도 테이블에서 관리한다.
  - 한 유저가 여러 학원에 소속될 수 있어야 한다.
  - `users.role`은 전역 role이고, 실제 학원 내 권한은 `academy_members.academy_role` 중심으로 관리한다.
- 학생과 선생님 모두 `status` 컬럼이 필요하다.
  - 예: `active`, `inactive`, `archived`, `deleted`
- hard delete보다 soft delete를 우선 고려한다.
  - 실제 삭제보다 상태 변경과 `updated_at` 기록을 우선 사용한다.
- 비밀번호 평문 저장은 금지한다.
  - 비밀번호는 `password_hash` 또는 Django `auth_user.password` 같은 해시 저장 구조만 허용한다.
- 연락처, 보호자 정보, 상담 정보는 개인정보/민감정보로 분리해 관리 여부를 명확히 한다.

## 6. 지금은 보류해도 되는 정보

아래 항목은 운영에 가치가 있지만, 지금 바로 핵심 테이블 컬럼에 넣기보다 별도 도메인 테이블이나 추후 기능으로 미루는 것이 안전하다.

- 학습 성향 상세 결과
- 시험별 점수 히스토리
- 취약 단원 상세 분석 이력
- 상담 로그
- 보호자 다중 연락처
- 리포트 발송 이력
- 알림 수신 세부 설정
- OCR 원본 메타데이터
- 문제 풀이 이미지/파일 메타데이터
- 출결 상세 이력
- 수업 녹화/자료 열람 로그
- 로그인 실패 잠금 정책 상세값

보류 이유는 다음과 같다.

- 이력성 데이터는 기본 엔티티 테이블에 넣으면 빠르게 비대해진다.
- 민감정보는 보관 필요성과 보관 기간을 먼저 정해야 한다.
- 운영하면서 실제 필요한 항목과 불필요한 항목이 갈릴 가능성이 높다.

## 7. 추천 엔티티 연결 구조

추천 연결 구조는 다음과 같다.

- `users`는 공통 계정 엔티티다.
  - 로그인 ID, 비밀번호 해시, 전역 role, 계정 상태를 가진다.
  - 기존 Django `auth_user`를 유지한다면 `users.auth_user_id`로 연결한다.
- `academies`는 학원 단위 엔티티다.
  - 학원 기본 정보와 대표 소유자 정보를 가진다.
- `academy_members`는 유저와 학원의 소속 관계를 관리한다.
  - 한 유저가 여러 학원에 속할 수 있게 해준다.
  - 실제 권한은 `academy_role` 중심으로 본다.
- `teacher_profiles`는 선생님 상세 프로필이다.
  - 계정과 분리된 표시명, 과목, 근무 형태를 저장한다.
  - 학원별로 프로필이 달라질 수 있으므로 `user_id + academy_id` 기준이 적합하다.
- `class_groups`는 반 엔티티다.
  - 반은 학원 소속이고, 담당 선생님은 `teacher_user_id`로 연결한다.
- `students`는 학생 기본 정보 엔티티다.
  - 학생 자체의 이름, 학교, 연락처, 상태를 가진다.
- `enrollments`는 학생과 반의 연결 엔티티다.
  - 학생이 여러 반을 수강할 수 있고, 상태 이력을 분리하기 좋다.

즉 관계를 문장으로 정리하면 다음과 같다.

- 한 `user`는 여러 `academy_members`를 가질 수 있다.
- 한 `academy`는 여러 `academy_members`, `teacher_profiles`, `class_groups`, `students`를 가진다.
- 한 `teacher_profile`은 하나의 `user`와 하나의 `academy`에 연결된다.
- 한 `class_group`은 하나의 `academy`, 하나의 담당 `teacher_user`를 가진다.
- 한 `student`는 하나의 `academy`에 속하고, 여러 `enrollments`를 통해 여러 반과 연결될 수 있다.

## 8. 최종 요약

- 선생님 필수 정보의 핵심은 `로그인 정보`, `학원 소속`, `학원 내 역할`, `표시명/담당과목`이다.
- 학생 필수 정보의 핵심은 `이름`, `학년`, `학교`, `소속 학원`, `반 연결`, `상태`다.
- 지금 바로 만들 핵심 테이블은 `users`, `academies`, `academy_members`, `teacher_profiles`, `class_groups`, `students`, `enrollments`다.
- 운영/보안 보조 테이블로 `auth_sessions`, `login_audit_logs`, `password_reset_tokens`를 둘 수 있다.
- 지금은 넣지 않아도 되는 정보는 시험 점수 히스토리, 상담 로그, OCR 메타, 학습 성향 상세 같은 이력성/민감 데이터다.
- 실제 DB 설계 시 우선순위는 다음 순서를 권장한다.
  - 1. `users`
  - 2. `academies`
  - 3. `academy_members`
  - 4. `teacher_profiles`
  - 5. `class_groups`
  - 6. `students`
  - 7. `enrollments`
  - 8. 운영/보안 보조 테이블

이 문서를 기준으로 다음 단계에서는 테이블별 컬럼명, 타입, FK, UNIQUE, INDEX, soft delete 정책을 확정하면 된다.
