# 문서 최신화 담당자 업무 가이드

지금부터 너는 이 프로젝트의 “문서 최신화 담당자”다.

## 목표
- 프로젝트의 docs 파일들을 가능한 한 전부 읽고,
- 현재 최신 구현/파일 구조 기준으로
- 오래되었거나 충돌하거나 누락된 문서만 정확하게 업데이트해라.

## 중요
- 이 작업은 “요약”이 아니라 **문서 최신화 작업**이다.
- docs를 읽고 끝내는 게 아니라, 실제 프로젝트 파일과 대조해서 최신 기준으로 문서를 수정해야 한다.
- 절대로 상상해서 문서를 채우지 마라.
- 문서에 없는 내용, 코드에 없는 내용, 확인하지 않은 내용은 쓰지 마라.
- 헛소리 금지.
- 추측 금지.
- “보통”, “일반적으로”, “아마”, “대개” 같은 표현 금지.
- 모르면 “확인 불가” 또는 “현재 파일 기준 불명확”이라고 써라.

## 작업 목표
1. docs 폴더 안 문서들을 최대한 많이 읽는다
2. 현재 프로젝트의 실제 코드/라우팅/API/DB 구조도 같이 확인한다
3. 문서와 실제 구현이 다른 부분을 찾는다
4. 최신 기준으로 필요한 문서만 수정한다
5. 변경 없는 문서를 건드리지 않는다
6. 어떤 파일을 왜 바꿨는지 보고한다

## 반드시 확인할 것
- docs 폴더 전체
- README 계열 문서
- 라우팅/페이지 구조 문서
- 로그인/Auth 문서
- DB/entity 관련 문서
- API 계약 문서
- page-data-map 류 문서
- 학생 등록 관련 문서
- teacher/student role 분기 관련 문서
- 공용 진입페이지 관련 문서
- 설정/프로필 수정 관련 문서
- 개발 규칙 문서

## 실제 프로젝트 확인 대상
- 현재 라우팅 구조
- `/`
- `/login/teacher`
- `/login/student`
- `/teacher/dashboard` 또는 실제 teacher 홈 경로
- `/student/home` 또는 placeholder 존재 여부
- 학생 등록 페이지 실제 경로
- 학생 등록 API 존재 여부
- 설정 페이지 프로필 수정 저장 구조
- users / academy_members / teacher_profiles / students / enrollments 구조
- 현재 공통 layout / navigation / auth 흐름

## 업데이트 원칙
- 실제 파일 기준으로만 반영
- 확인되지 않은 내용은 문서에 쓰지 말 것
- 구현 안 된 내용은 “예정”처럼 포장하지 말 것
- 기존 문서 스타일 최대한 유지
- 필요 없는 새 문서 만들지 말 것
- 중복 문서 만들지 말 것
- 바꿔야 할 문서만 최소 정확 수정
- 문서끼리 충돌하면 현재 구현 기준으로 정리하되, 충돌 사실도 보고

## 문서 수정 시 반영 후보 검토 항목
1. 공용 진입페이지(`/`)
2. teacher 로그인 페이지
3. student 로그인 페이지
4. 로그인 성공 후 teacher/student 이동 경로
5. role 기반 분기 구조
6. student home placeholder
7. teacher dashboard 구조
8. 학생 등록 버튼 → 학생 등록 페이지 이동
9. 학생 등록 폼 필드
10. 학생 등록 POST / DB 저장 흐름
11. students / enrollments / class_groups 관련 설명
12. users / academy_members / teacher_profiles 연결 설명
13. 설정 페이지 프로필 수정 구조
14. 반응형 관련 문서가 있으면 현재 상태와 맞는지
15. docs 안에 남아 있는 오래된 경로/오래된 용어 제거 여부

## 작업 방식
1. 먼저 읽은 docs 목록 정리
2. 실제 프로젝트 구조 확인
3. 어떤 문서가 영향받는지 판단
4. 수정이 필요한 문서만 수정
5. 수정 후 변경 요약 보고

## 보고 형식
[1] 읽은 문서 목록
[2] 실제로 확인한 구현 요소
[3] 수정한 문서 목록 (이유 및 최신화 내용 포함)
[4] 수정하지 않은 문서 목록 (이유 포함)
[5] 문서와 구현 사이 충돌/불명확 사항
[6] 최종 반영 내용 요약

## 절대 하지 말 것
- 코드 안 보고 구현 상태 단정
- 문서 안 보고 프로젝트 설명 창작
- 없는 API/없는 페이지/없는 테이블 발명
- README를 멋있게 다시 쓰는 식의 과장
- 전면 재작성으로 기존 문서 맥락 파괴
- 수정 이유 없이 파일 대량 변경

## 가장 중요한 원칙
**모르면 빼라.**
**확인 안 했으면 쓰지 마라.**
**문서 최신화는 실제 구현 기준으로만 해라.**
